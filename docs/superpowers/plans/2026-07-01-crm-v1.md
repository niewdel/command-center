# CRM v1 Implementation Plan

> **For agentic workers:** execute task-by-task (subagent-driven). Steps use `- [ ]`. Build on branch `feat/crm-v1`. Keep the build GREEN (`npx tsc --noEmit` + `npm test` + `npm run build`). Do NOT deploy to production.

**Goal:** A generic, multi-tenant (RLS-isolated) CRM inside Command Center — Companies (accounts) with multiple Contacts, a drag-drop pipeline board, company detail with activity timeline + tasks, and a seeded Demo tenant. Foundation for Proposals + Billing.

**Architecture:** Tenant = workspace. Extend the existing `clients` table (= Companies) additively; new `contacts`/`crm_activities`/`crm_tasks` tables carry denormalized `workspace_id` + RLS. Server-side data access via the app's service-role + workspace-scoping pattern. On-brand v3 UI (Jet/Onyx/Blue, Montserrat/Inter, pill CTAs, blue-dot bullets) reusing existing `globals.css` tokens.

## Global Constraints
- Brand v3 per `docs/superpowers/research/2026-07-01-brand-v3-reference.md`. Voice: outcome-first, no em-dashes, no AI slop. Reuse existing design tokens/utilities, don't redefine.
- Multi-tenant: every CRM row workspace-scoped; RLS on every new table; no cross-tenant reads. No Niewdel-specific hardcoding (generic product).
- TypeScript strict, no `any`. API routes: verify session, scope by workspace, Zod validation, structured errors.
- `npx tsc --noEmit`, `npm test`, `npm run build`, `eslint` all clean before each commit. Conventional commits.
- Research (context7/official docs), don't guess: Supabase RLS, @dnd-kit for the board, Next.js 16 App Router patterns.
